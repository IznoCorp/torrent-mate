# Design: Test Coverage & Design-Contract Testing

**Date**: 2026-05-08
**Status**: Approved — awaiting implementation
**Scope**: Entire project (`personalscraper`)
**Target**: 90% unit + integration branch coverage, enforced in CI

## 1. Purpose

Raise test coverage from the current 44% to 90% while ensuring tests are useful — not
coverage padding. Every integration test maps to a documented behavior in a DESIGN.md
or reference doc, creating traceability from documentation through tests to code.

## 2. Current State

- **Default suite**: 411 tests (370 unit + 21 integration), runs in ~5s via `make test`
- **Full repo**: ~2900 tests, most are manual E2E (deselected by markers)
- **Coverage**: 43.77% — well below the 80% `fail_under` declared in `pyproject.toml`
- **CI**: `.github/workflows/ci.yml` already runs `--cov-fail-under=80` in the `test` job.
  This job currently fails on `main` (coverage is 44%). The existing CI coverage step will
  be replaced by the new `test-cov` job described in this design.
- **Biggest gaps**: `trailers/` (0-52%), `scraper/tv_service.py` (15%), `scraper/trailer_finder.py` (27%), `scraper/ytdlp_downloader.py` (34%), `verify/fixer.py` (26%)

## 3. Architecture — Four Pillars

### Pillar 1 — Feature-Anchored Coverage Cycles

Each major feature gets its own test improvement cycle:
audit coverage → write design-contract integration tests per the feature's DESIGN.md →
fill unit gaps → verify global coverage rose.

Coverage is measured **globally** (`--cov=personalscraper`). Individual feature cycles
do not have per-module thresholds — success is measured by the global `fail_under` bump.
This keeps measurement simple and avoids fragile per-module targets.

### Pillar 2 — Design-Contract Integration Tests

Each integration test maps to a specific section in a design or reference doc via
docstring markers. The marker uses a **stable heading anchor** derived from the section
title using the same algorithm as GitHub: lowercase, strip all non-alphanumeric
characters (except spaces and hyphens), replace spaces with hyphens, collapse multiple
hyphens, strip leading/trailing hyphens. This ensures the marker anchor and the
`audit_design_coverage.py` script produce identical output.

```
Design: docs/features/api-unify/DESIGN.md#circuit-breaker-open-after-3-failures
Contract: Circuit breaker opens after 3 consecutive failures and rejects
subsequent requests with CircuitBreakerOpenError until the reset timeout.
```

When a section heading changes, only the `Design:` line in the test docstring needs
updating — test function names are unaffected.

A test is "design-contract" if and only if it has both `Design:` and `Contract:`
markers in its docstring. Tests without both markers are regular tests. Markers are
only recognized on function/method docstrings, not module-level docstrings — the
`update_feature_map.py` script only parses `ast.FunctionDef` nodes.

**Tier rule**: Design-contract tests prefer integration tier (real module interactions
with mocked external services via local test servers or dependency injection). When a
contract inherently requires a live qBittorrent instance or real API keys, the test
must live in `tests/e2e/` (or a subdirectory thereof) and be marked `e2e_torrent` or
`roundtrip`. CI excludes these via `--ignore=tests/e2e`. Such contracts are still
tracked in the feature map for documentation traceability, but they are excluded from
the `fail_under` coverage gate.

### Pillar 3 — Per-Feature Mapping Files

`tests/feature_map/<codename>.json5` — one file per feature, mapping each design section
to its test functions. Per-feature files eliminate merge conflicts between unrelated
feature branches.

**Codename derivation rule**: The codename is the directory name immediately following
`features/` in the design doc path.

- `docs/features/api-unify/DESIGN.md` → codename `api-unify`
- `docs/reference/architecture.md` → codename `architecture`
- `docs/archive/features/trailer/DESIGN.md` → codename `trailer`

If no `features/` segment exists, the document's stem (filename without `.md`) is used.

Example map file:

```json5
// tests/feature_map/api-unify.json5
{
  feature: "api-unify",
  design: "docs/features/api-unify/DESIGN.md",
  sections: {
    "#circuit-breaker-open-after-3-failures": {
      tests: [
        "tests/integration/test_design_api_transport.py::test_circuit_breaker_opens_after_3_failures",
      ],
    },
  },
  // Sections intentionally excluded from the coverage audit (untestable content
  // like "Non-Goals", "Purpose", "Future Enhancements", "Current State"):
  skip_audit: [
    "#purpose",
    "#current-state",
    "#non-goals",
    "#future-enhancements-post-80",
  ],
}
```

Generated by `python3 scripts/update_feature_map.py` scanning docstring `Design:` markers.
The script is run via a **git pre-commit hook** (at `hooks/pre-commit`) that
auto-regenerates the appropriate map file when `test_design_*.py` files are staged.
This gives instant feedback at commit time rather than waiting for CI.

### Pillar 4 — Staged Coverage Thresholds

`fail_under` in `pyproject.toml` is lowered from the current 80 to the real baseline
(measured after enabling `--cov-branch`) and then bumped progressively as each feature
cycle lands. This prevents backsliding while avoiding one giant gate that never passes.

Branch coverage (`--cov-branch`) is enabled from the start. Branch coverage catches
half-tested conditionals (e.g., testing the "open" path of a circuit breaker but not
the "half-open" or "reset" path). Adding it later would require re-baselining.

**Note**: Enabling `--cov-branch` may lower the measured percentage compared to the
current 43.77% (which is measured without branch coverage). The baseline is
re-measured after enabling `--cov-branch` in Task 1.1 of the plan. The staged
threshold progression adjusts from that rebaselined value.

The progression (after rebaselining): initial → 50% → 60% → 70% → 80% → 85% → 90%.

## 4. Feature Cycle Order

| #   | Feature                                                | Design Docs                                                             | Rationale                                                                                                                |
| --- | ------------------------------------------------------ | ----------------------------------------------------------------------- | ------------------------------------------------------------------------------------------------------------------------ |
| 1   | **api-unify** (`personalscraper/api/`)                 | `docs/features/api-unify/DESIGN.md`                                     | Active branch, 27 phases done. Builds the template.                                                                      |
| 2   | **Scraper** (`personalscraper/scraper/`)               | `docs/reference/scraping.md`                                            | Core pipeline — wrong metadata poisons downstream. Critical gaps at `tv_service.py` (15%) and `trailer_finder.py` (27%). |
| 3   | **Dispatch + Verify**                                  | `docs/reference/storage.md`, `docs/reference/pipeline-internals.md`     | Movement to permanent storage — mistakes hard to undo. Already well-covered, shorter cycle.                              |
| 4   | **Trailers** (`personalscraper/trailers/`)             | `docs/reference/trailers.md`, `docs/archive/features/trailer/DESIGN.md` | Worst coverage (0% CLI, ~30% state/scanner). Pipeline-critical (blocks on error).                                        |
| 5   | **Indexer** (`personalscraper/indexer/`)               | `docs/reference/indexer.md`, `docs/reference/indexer-json-shapes.md`    | Large and complex. Decently tested. Deep domain knowledge needed for remaining gaps.                                     |
| 6   | **Remaining** (sorter, ingest, process, library, conf) | `docs/reference/architecture.md` + relevant reference docs              | Mixed bag. Each small, some already 85%+. Cleanup pass.                                                                  |

Each cycle bumps `fail_under` by one increment. Cycles on existing features (scraper,
trailers, indexer, etc.) do **not** require new feature branches or `/implement:feature`.
They follow a simplified workflow: audit → add tests directly → bump threshold → PR.
The full `/implement:feature` lifecycle is only used when a cycle coincides with active
feature development (as with api-unify).

## 5. CI Enforcement (GitHub Actions)

### Coverage Job

Replaces the existing `test` job's coverage step in `.github/workflows/ci.yml`:

- Runs `make test-cov` as a single test pass (unit + integration with `--cov-branch` and `--cov-fail-under=N`)
- Threshold is read from `pyproject.toml` via a helper that falls back to `tomli` on Python 3.10:
  `THRESHOLD := $(shell python3 -c "try: import tomllib\nexcept ImportError: import tomli as tomllib\nprint(tomllib.load(open('pyproject.toml','rb'))['tool']['coverage']['report']['fail_under'])")`
- The `test-cov` target generates both `--cov-report=term` (human-readable) and `--cov-report=xml` (for codecov/codecov-action)
- Failure is a **hard CI block** — same as lint or mypy errors

### Design-Gaps Job

A separate CI job that:

1. Downloads the `.coverage` artifact from the coverage job via `actions/download-artifact`
2. Runs `python3 scripts/audit_design_coverage.py --strict` (warnings by default, promoted to `--strict` at 80% global threshold)

### CI Workflow Order

```
lint (ruff + mypy)  ──┬── test-cov (single pass, coverage enforced, upload .coverage artifact)
                      └── (lint and test-cov run in parallel)
                              │
                              └── design-gaps (download artifact, audit orphan sections)
```

### Local-Only Targets

- `make test-unit` — fast feedback for TDD, excludes integration and E2E, no coverage
- `make test-integration` — integration suite only, no coverage
- `make test-cov` — full coverage run (what CI uses)

## 6. Design-Contract Test Format

### File Naming

`tests/integration/test_design_<module>.py` — placed alongside existing integration tests.

### Structure Rules

1. One test method = one contract clause. If a design section makes 3 claims, that's 3 tests.
2. Test method is named by behavior, not section number:
   `test_circuit_breaker_opens_after_3_failures` (NOT `test_s4_5_...`).
3. Mandatory `Design:` and `Contract:` markers in the function/method docstring (not the module or class docstring). The `Design:` marker uses
   a stable heading anchor, not a section number.
4. Prefer integration tier with mocked external services. Contracts requiring live services
   are placed in `tests/e2e/` and marked `e2e_torrent` or `roundtrip` — excluded from `fail_under` gate.

### Test Quality Criteria

Design-contract tests must be:

- **Behavioral, not structural**: verify observable outcomes, not internal call patterns or log messages
- **Minimally mocked**: mock at module boundaries (I/O, external APIs), not within the module under test
- **Single concern**: one assertion family per test. Verifying "circuit breaker opens AND returns specific error" is fine. Verifying "circuit breaker opens AND cache eviction fires" is two tests.
- **Readable as specification**: the test name and assertions should be understandable without reading the design doc

These criteria are enforced during code review. The `/implement:check` skill's 7th check
verifies that design-contract tests exist and pass; review enforces that they are _good_.

### Fixture Strategy

Design-contract tests use the existing `tests/integration/conftest.py` fixtures
(temporary staging directory, patched Config, mock servers). Additional feature-specific
fixtures go in a `conftest.py` alongside the test files, following the existing
convention of importing from the nearest `conftest.py` in the hierarchy.

### Example

```python
def test_circuit_breaker_opens_after_3_failures(staging_dir, mock_api_server):
    """Circuit breaker opens after 3 consecutive failures.

    Design: docs/features/api-unify/DESIGN.md#circuit-breaker-open-after-3-failures
    Contract: Circuit breaker opens after 3 consecutive failures and rejects
    subsequent requests with CircuitBreakerOpenError until the reset timeout.
    """
    ...
```

## 7. Feature-to-Test Mapping

### Location and Format

`tests/feature_map/<codename>.json5` — one file per feature, no monolithic file.

### Generation

`python3 scripts/update_feature_map.py` scans `tests/` for `Design:` markers on
function/method docstrings (not module or class docstrings) and regenerates the
appropriate per-feature map files. It is invoked via a **git pre-commit hook**
(`hooks/pre-commit`) that fires when `test_design_*.py` files are staged. This gives
instant feedback at commit time rather than waiting for CI.

### Excluding Untestable Sections

Design doc sections that describe non-functional content (Purpose, Current State,
Non-Goals, Future Enhancements, Constraints) are inherently untestable. The map file
format includes a `skip_audit` key listing these anchors. The `audit_design_coverage.py`
script skips them. When a feature's brainstorm phase creates a stub map file, common
untestable anchors are pre-populated.

### CI Checks

| Check            | Severity        | Description                                                                                                      |
| ---------------- | --------------- | ---------------------------------------------------------------------------------------------------------------- |
| Orphan sections  | Warning → Error | Design sections with zero tests. Warning initially, promoted to error once 80% global coverage is reached.       |
| Stale references | Warning         | Test markers pointing to anchors not found in any design doc. Means a test references a deleted/renamed section. |

### Maintenance

Run `python3 scripts/update_feature_map.py` before committing new design-contract tests
(the pre-commit hook does this automatically). Every 6 months, audit `Design:` markers
for correctness against current design docs as part of routine maintenance.

## 8. Tooling & Scripts

### `scripts/update_feature_map.py`

Scans `tests/` for `Design:` docstring markers on function/method nodes (via `ast.parse`)
and regenerates `tests/feature_map/*.json5`. Parses the `Design:` line to extract the
design doc path and anchor name. Groups by feature codename using the rule in §3 Pillar 3.
Outputs one JSON file per feature (`.json5` extension, standard JSON content).

Marker regex: `^Design:\s*(\S+)#(.+)$` (multiline). The `Design:` line must be at the
start of a docstring line (after `inspect.cleandoc` normalization). The design path is
the full path relative to repo root. The anchor is everything after `#`.

### `scripts/audit_design_coverage.py`

Parses design docs (Markdown) to extract `##`, `###`, and `####` headings and their
GitHub-style anchors. The anchor algorithm must match the one described in §3 Pillar 2:
lowercase → strip non-alphanumeric (except spaces/hyphens) → replace spaces with hyphens →
collapse multiple hyphens → strip leading/trailing hyphens.

Searches for design docs in `docs/features/*/DESIGN.md`, `docs/reference/*.md`, AND
`docs/archive/features/*/DESIGN.md` to cover archived feature design docs.

When two headings produce the same anchor, appends `-1`, `-2`, etc. (matching GitHub's
algorithm, which starts at `-1` for the first duplicate).

Compares against `tests/feature_map/*.json5` to list sections with zero tests,
excluding any anchors listed in the `skip_audit` key of each map file. Used at the
start of each feature cycle to scope the work.

Exit codes: 0 on success (no gaps or no `--strict`), 1 if gaps found with `--strict`.

### `hooks/pre-commit` (git pre-commit hook)

Bash script that fires when `test_design_*.py` files are staged. Runs
`python3 scripts/update_feature_map.py` and stages the updated map files.
Named `pre-commit` (not `update_feature_map.py`) because git hooks are defined
by their git hook type name.

### New Makefile Targets

```makefile
THRESHOLD := $(shell python3 -c "try: import tomllib\nexcept ImportError: import tomli as tomllib\nprint(tomllib.load(open('pyproject.toml','rb'))['tool']['coverage']['report']['fail_under'])")

test-unit:
	python3 -m pytest tests/ --ignore=tests/integration --ignore=tests/e2e -q

test-integration:
	python3 -m pytest tests/integration/ -q

test-cov:
	python3 -m pytest tests/ --ignore=tests/e2e -q --no-header \
		--cov=personalscraper --cov-branch --cov-report=xml --cov-report=term --cov-fail-under=$(THRESHOLD)
```

The `python3` interpreter is used in the THRESHOLD derivation and test targets for
portability (ubuntu-latest CI runners use `python3`). The existing `Makefile` convention
of `python` is replaced in these new targets.

Note: `-n auto` (parallel execution) is intentionally omitted from `test-cov` to avoid
coverage data corruption from concurrent `.coverage` file writes.

## 9. Integration with `/implement:feature` Workflow

| Phase                   | Coverage Action                                                                                                                                                               |
| ----------------------- | ----------------------------------------------------------------------------------------------------------------------------------------------------------------------------- |
| `/implement:brainstorm` | Stub map file created at `tests/feature_map/<codename>.json5` with section anchors listed, empty test arrays, and common untestable anchors in `skip_audit`. Declares intent. |
| `/implement:plan`       | Each phase file lists which DESIGN.md sections it implements and estimated test count.                                                                                        |
| `/implement:sub-phase`  | Design-contract tests written alongside implementation code. Same commit. Pre-commit hook auto-updates the map file.                                                          |
| `/implement:check`      | 7th check added: design-contract coverage. For each section a phase claims, at least one matching test must exist and pass.                                                   |
| `/implement:feature-pr` | CI coverage job enforces `fail_under`. PR blocked if coverage drops below threshold.                                                                                          |
| Merge                   | `fail_under` bumped if cycle warrants it.                                                                                                                                     |

**For coverage-only cycles on existing features** (no active feature branch), skip
`/implement:brainstorm` and `/implement:plan`. Work directly: audit gaps with
`audit_design_coverage.py`, add tests, make a PR with the bumped threshold. A stub
map file is created manually at the start of the cycle if one does not already exist.

### New-Code Coverage Policy

New code in a PR must maintain or improve the module's global coverage. If a module has
low coverage, new code added to it must be accompanied by tests that improve its
contribution to the global percentage. This is enforced by the global `fail_under` gate
in CI — adding untested code to a low-coverage module will cause the `fail_under` check
to fail. For modules where a stricter per-module check is desired, `--cov-fail-under`
can be scoped to that module in a future enhancement.

### Duplicate Test Policy

When a new design-contract test overlaps with an existing legacy test, both are kept.
Existing tests are preserved unless they actively test wrong behavior (see Non-Goals).
Duplication between a design-contract test and a legacy test is acceptable — the
design-contract test provides traceability and documentation value that the legacy test
may lack.

## 10. Constraints & Non-Goals

### Constraints

- Coverage measured on **unit + integration only** (default suite, excluding E2E). E2E tests require live qBit/APIs that CI runners don't have.
- 90% target applies to unit + integration coverage with branch coverage enabled. No separate E2E coverage target.
- `personalscraper/__main__.py` is excluded from coverage (already configured).
- `-n auto` is disabled in coverage runs to prevent data corruption.
- Test quality criteria (behavioral, minimally mocked, single concern, readable) are enforced during code review.

### Non-Goals

- 100% coverage — the last 10% is diminishing returns (defensive branches that never trigger, `__repr__`, etc.)
- Rewriting or deleting existing tests — existing tests are preserved unless they actively test wrong behavior
- Modifying E2E test infrastructure — the E2E suite is already comprehensive (2600+ tests)
- Retroactively adding `Design:`/`Contract:` markers to existing tests — only new tests written as part of feature cycles carry markers
- Phantom-path detection (the original `coverage_gap_report.py`) — demoted to future enhancement. Requires design-section-to-source-code-line mapping that does not exist yet. Revisit after 80% coverage is reached

### Future Enhancements (post-80%)

- `scripts/coverage_gap_report.py` — detect sections whose tests exist but don't exercise the target code paths. Requires mapping design sections to source file/line ranges, scoped after 80% global coverage is achieved.
