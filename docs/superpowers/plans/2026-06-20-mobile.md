# mobile — Responsive Mobile UI Implementation Plan

> **For agentic workers:** REQUIRED SUB-SKILL: Use superpowers:subagent-driven-development (recommended) or superpowers:executing-plans to implement this plan task-by-task. Steps use checkbox (`- [ ]`) syntax for tracking.

**Goal:** Make the whole KanbanMate web UI usable on phones — hamburger-drawer nav, list→detail master-detail, fluid panels, sheet dialogs — without changing the desktop layout or any backend.

**Architecture:** A reactive `useIsMobile()` hook (matchMedia 768 px) drives a mobile layout branch in the SPA. The desktop sidebar nav is extracted into a shared `SidebarNav`, reused inside a slide-in drawer on mobile. A `MobileMasterDetail` helper turns the side-by-side panels into list→full-screen-detail+back on mobile. Everything else is fluid CSS.

**Tech Stack:** React 18 + the in-repo `kanbanmate-design` system + the existing i18n. No new dependency. **Frontend-only** (`web/`), no backend/daemon change.

## Global Constraints

- **Frontend-only**: no change to `src/kanbanmate/**`, no backend/endpoint/daemon change, no new dependency.
- **Desktop unchanged**: at viewport ≥ 769 px the rendered layout must be identical to today.
- Breakpoint: **`(max-width: 768px)`** = mobile.
- Every user-facing string via `t()` in `web/src/i18n/{en,fr}.yaml` (English default + fallback).
- No JS test runner in this repo → each task verifies with `npm --prefix web run build` (must succeed) + a **manual** narrow-viewport check (browser devtools responsive mode, or `km.iznogoudatall.xyz` on a phone). This matches the SPA-light testing convention used by bridge/monitoring.
- Conventional Commits, scope `mobile`. No version prefixes / AI attribution. (No SemVer bump — frontend-only, but the feature ships under the existing wheel; bump to 0.10.1 at PR time if desired.)
- `webui/` build output stays gitignored (CI rebuilds; packaged under `[ui]`).

## Task 1: `useIsMobile` hook

**Files:**

- Create: `web/src/useIsMobile.js`

**Interfaces:**

- Produces: `export default function useIsMobile(): boolean` — `true` when `matchMedia("(max-width: 768px)")` matches; re-renders on viewport crossing.

- [ ] **Step 1: Implement the hook**

```js
// web/src/useIsMobile.js
// Reactive viewport check for the mobile layout branch (responsive mobile, DESIGN §3).
// 768px = phones + small tablets portrait. Re-renders when the viewport crosses the breakpoint.
import React from "react";

const QUERY = "(max-width: 768px)";

export default function useIsMobile() {
  const get = () =>
    typeof window !== "undefined" && window.matchMedia
      ? window.matchMedia(QUERY).matches
      : false;
  const [isMobile, setIsMobile] = React.useState(get);
  React.useEffect(() => {
    const mq = window.matchMedia(QUERY);
    const onChange = () => setIsMobile(mq.matches);
    mq.addEventListener("change", onChange);
    onChange(); // sync in case it changed between render and effect
    return () => mq.removeEventListener("change", onChange);
  }, []);
  return isMobile;
}
```

```jsx
// web/src/components/MobileMasterDetail.jsx
// Mobile master-detail: full-width list, tap → full-screen detail + back header (DESIGN §4.2).
import React from "react";
import { useT } from "../i18n/index.jsx";

export default function MobileMasterDetail({
  selected,
  onBack,
  backLabel,
  list,
  detail,
}) {
  const { t } = useT();
  if (!selected) return list;
  return (
    <div>
      <button
        onClick={onBack}
        style={{
          display: "flex",
          alignItems: "center",
          gap: 6,
          border: "none",
          background: "transparent",
          cursor: "pointer",
          color: "var(--foreground)",
          fontSize: 14,
          padding: "4px 0 12px",
        }}
      >
        ← {backLabel || t("common.back")}
      </button>
      {detail}
    </div>
  );
}
```
