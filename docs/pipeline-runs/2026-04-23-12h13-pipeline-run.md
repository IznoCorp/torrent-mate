### Patterns systémiques

1. **Tracker dual-location (bug #1)** : `Settings.data_dir` (legacy, CWD-relative) vs `Config.paths.data_dir` (json5, absolu). Le tracker écrit au mauvais endroit, générant deux fichiers divergents. Symptôme pour d'autres chemins probable (à vérifier).

2. **Propagation rename manquante (bug #10)** : reclean modifie le staging mais ne propage pas sur disques. Si une série existe déjà sur disque sous un nom obsolète, dispatch crée un doublon. La couche manquante est un « rename-on-disk hook » déclenché par reclean OU l'utilisation d'un ID stable (TMDB/TVDB) pour le matching dispatch.

3. **Bugs NFO mineurs (bugs #4, #5)** : scraper écrit `None` littéral dans `<uniqueid>` épisode, et `default="true"` sur un ID=0 quand TMDB n'a pas la série. Plex/Kodi/Jellyfin en pâtira sur lookups.

4. **Observabilité dispatch (bugs #11, #12)** : dry-run manque de détails critiques (action type, état index) pour valider avant rsync destructif.

### Cascades

- Bug #1 (tracker wrong location) → cascade sur state-validator qui lit le mauvais fichier et rapporte de faux phantoms (bug #3).
- Bug #10 (reclean sans rename disque) → bloquerait le dispatch auto sans intervention humaine.
- Issue B (NFC/NFD) apparue dans l'analyse Python mais **pas** en réalité filesystem → démontre que les checks side-band peuvent générer des fausses alertes ; `dispatcher.py` est fiable sur ce point.

### Bug #9 — verify --fix spurious deprecation → TRAITÉ (`f84863e`)

Diagnostic : `cli.py:279` — `typer.Option(True, …)` sur un bool défaut True ⇒ impossible de distinguer « --fix omis » vs « --fix explicite ». Le guard `if fix:` firait toujours.

Fix : passer à `Optional[bool]` avec défaut `None`, tester `is not None` pour le warning, et convertir en bool False si omis (par défaut, pas de fix — conforme à la dépréciation).

Tests : `personalscraper verify` ne produit plus de warning. Les 82 tests `nfo_generator` passent.

### Bug #4 + Bug #5 — NFO uniqueid=None / ID=0 default=true → TRAITÉ (`a53a44f`)

Diagnostic :
- `nfo_generator.py:321-328` (épisode) : `str(episode_data.get("tvdb_id", ""))` produit `"None"` quand la clé existe mais la valeur est `None` (le default du `.get` ne s'applique pas).
- `nfo_generator.py:216-230` (show) : `default="true"` forcé sur TMDB même si `id=0` ou absent.

Fix unifié : construire les uniqueids seulement si la valeur est réellement utilisable (`not in (None, 0, "0", "", "None")`). Promouvoir le provider alternatif en `default="true"` quand le provider préféré est inutilisable. TMDB reste default pour les shows, TVDB pour les épisodes — fallback seulement en repli.

Test impact : `SAMPLE_EPISODE_DATA` avait `tvdb_id=""` ce qui suppose un tag vide — remplacé par une valeur valide `"9876543"`. Les 82 tests passent.
