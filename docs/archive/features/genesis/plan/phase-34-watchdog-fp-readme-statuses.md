# Phase 34 — Watchdog leaked-thread false-positive fix + README status table

Phase 34 fixes a live false positive and documents the dashboard health pill. **Fix:** `app/tick.py`'s `_watchdog_executor` used to warn `"tick exiting with N worker thread(s) still running (abandoned hung action?)"` on EVERY tick that ran an action, because it counted `t.is_alive()` on the pool's `_threads` and a `ThreadPoolExecutor` worker stays alive IDLE (parked on the work queue) between tasks — so a successful action tripped the warning even with failures=0. The fix introduces an authoritative per-tick timeout registry (`_TIMED_OUT_ACTIONS`, a `WeakKeyDictionary` keyed on the executor): the three watchdog wrappers (`_run_with_watchdog`, `_run_value_with_watchdog`, `_run_callable_with_watchdog`) record an action's description there only when `future.result(timeout=…)` actually raises `TimeoutError` (a genuinely orphaned worker), and `_watchdog_executor` warns at exit ONLY when that list is non-empty, naming the abandoned action(s); idle workers get a tiny grace join (`_IDLE_WORKER_GRACE_S = 0.2s`) post-shutdown purely cosmetically, and the non-blocking `shutdown(wait=False, cancel_futures=True)` semantics are unchanged. Tests assert a completed action logs no leak warning and a timed-out action logs a warning naming it. **Docs:** README.md gains a "Project status (dashboard health pill)" section with a table for INACTIVE / OFF_TRACK / AT_RISK / ON_TRACK / COMPLETE — each row's trigger conditions mirror `core/status_update.py::compute_status` precedence (PAUSE → INACTIVE; block/gate_fail or Blocked agent → OFF_TRACK; waiting/reap/rate_limit/queue-over-cap/stale heartbeat → AT_RISK; idle → COMPLETE; else ON_TRACK) plus the operator action (e.g. AT_RISK+waiting → `tmux attach -t ticket-<n>`; INACTIVE → `kanban resume`).
