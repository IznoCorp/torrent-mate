# Phase 35 — E2E findings fixes (probe / markers / doctor)

Three independent fixes observed live during end-to-end orchestration runs, one commit each. (1) `GitWorktreeWorkspace.has_unpushed_work` was false-positiving every Done-arrival reclaim because the launch step `ensure_manual_merge_mode` (adapters/perms.py) rewrites the `**PR merge**:` line of IMPLEMENTATION.md at every launch, so an idle worktree always shows ` M IMPLEMENTATION.md`; the probe now diffs that single-file dirtiness and ignores it iff the only changed content lines are `**PR merge**:` pin lines, while any other path or any other diff content still counts as real unpushed work (probes stay timed + fail-closed-to-False). (2) `core/launch_keys.py::REPL_READY_MARKERS` missed claude v2.1.170's chrome (bare `❯` prompt + `auto mode on (shift+tab to cycle)` footer), so every launch's readiness poll timed out (20×0.5s) and sent late (+10s); the bare `❯`, `auto mode on`, and `shift+tab to cycle` markers are now READY signals, leaving the trust markers and `is_waiting_for_input` (where a bare `❯` stays NOT-waiting) untouched. (3) A new `kanban doctor` check (`_check_helper_shims`) derives every `kanban-*` console script from `importlib.metadata.entry_points(group="console_scripts")` and asserts each resolves via `shutil.which`, FAILing with the hint `run: pip install -e . (stale editable install)` — closing the gap the §29 `kanban-update-body` missing-shim incident exposed (the prior gate test only checked declaration in `[project.scripts]`, not runtime resolution). Tests cover each branch; `make check` green before each commit.
