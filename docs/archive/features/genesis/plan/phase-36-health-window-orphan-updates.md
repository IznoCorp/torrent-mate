# Phase 36 — Health window + orphan status updates

Two small dashboard fixes observed live, one commit each. (1) `core/status_update.py::compute_status` flagged OFF_TRACK/AT_RISK on ANY `block` / `gate_fail` / `reap` / `rate_limit` entry in the recent-events ring, but the ring (last ~10, `app/status_reporter` + store) has no time decay, so on a quiet board a morning block pinned the pill OFF_TRACK for hours (observed live: 3 stale #151 blocks → OFF_TRACK all afternoon); a new `EVENT_HEALTH_WINDOW_S = 3600.0` constant now gates the event-driven verdict so only entries with `state.now - e.ts <= window` count toward the pill (boundary inclusive), while older events still RENDER in the "Événements récents" list — the agents-based conditions (waiting, Blocked-parked, stale heartbeat, queue>cap) and the idle→COMPLETE branch are untouched. (2) When `report_status`'s `update_status_update` of a stale id fails and it re-creates the rolling pill (`app/status_reporter.py`), the OLD GitHub status update lingered (observed: 3 stacked on the v2 project); a new `delete_status_update(id)` capability — a pure `deleteProjectV2StatusUpdate` builder in `_queries.py` mirroring `update_status_update`, the `GithubClient` method, the `ProjectStatusReporter` port method, and a `_NullStatusReporter` no-op — is now invoked best-effort after a successful re-create to delete the orphan, fail-soft (a delete failure is logged and swallowed). Tests cover each: an old block → ON_TRACK, a fresh block → OFF_TRACK, the inclusive boundary, an aged degraded event; the re-create path deletes the stale id and a delete failure is swallowed; the adapter issues the mutation and raises on a GraphQL error. `make check` green before each commit.
