## Per-slice summaries

### TRANSITION/DISPATCH model — per-(from,to) whitelist + rollback + per-transition action vs NEW per-destination-column class decide()
NEW is structurally UNFAITHFUL to the PoC's transition model — this is the operator's headline gap, and it is a genuine feature_loss cluster, not a pivot artifact. The PoC dispatches on a per-(from_col, to_col) WHITELIST loaded from transitions.yml (transitions.py + dispatch.py): a present pair runs its configured action (prompt and/or script), an absent pair is REJECTED and the card is ROLLED BACK to from_col, and wildcards (*→to, from→*) with explicit-wins precedence model parking/Cancel rows. Each transition carries its own prompt, script (mechanical gate or LLM gate), profile, permission_mode (validated, bypass-banned), advance ("stop"|"auto:<col>"), and on_fail (""|move:<col>|rollback). decide_transition returns FOUR verdicts (launch/run_script/noop/rollback) and the runner adds skip/queue/block/teardown/reset. NEW (core/decide.py + core/columns.py) throws ALL of this out and replaces it with a per-DESTINATION-COLUMN class model: a Transition is (ticket, from_column, to_column) and decide() resolves ONLY the destination column's class (AGENT/REACTIVE/INERT) — from_column is used solely for the Cancel→Backlog reset rule. There is NO whitelist, NO rollback verdict, NO per-transition prompt/script/on_fail/advance, and NO run_script verdict. Worse: NEW's columns.yml.tmpl and DESIGN §8 advertise per-column prompt/permission_profile/interactive_only, but core/columns.py::load_columns (lines 78-89) parses ONLY key/name/triggers_agent/action and SILENTLY DROPS prompt/permission_profile/interactive_only; the Column dataclass (domain.py 34-46) has no fields for them. The launch prompt is a single GLOBAL agent_command (default "claude", wiring.py:56), identical for every agent column — so the PoC's per-step /implement:* prompt routing is not actually wired anywhere. None of these drops relate to the n8n/webhook/HMAC pivot; they are unauthorised drift from the source-of-truth PoC. The permission_mode validation/ban logic survives in spirit but moved to perms.py for the two static profiles (safe/trusted), not per-transition.

### RUNNER / orchestration loop (PoC runner.py + cli_dispatch.py + audit.py vs NEW app/tick.py, app/actions.py, app/wiring.py, daemon/loop.py)
The genesis pivot (n8n/webhook/HMAC ingress → in-daemon polling, launchd-reaper → in-tick reaper) is faithfully and intentionally reflected: signature verify, delivery-id idempotency, recent-bot-move dedup, in-flight lock, and the whole webhook payload path are gone by design. BUT the runner slice also LOST a large amount of orchestration capability that has nothing to do with the ingress pivot, and these are the operator-relevant drifts. The PoC's pure decision model carried prompt/script/profile/on_fail/advance/permission_mode per transition and four action verdicts beyond launch/teardown/reset/noop (run_script, rollback, queue, block-via-misconfig); NEW collapsed this to a 5-kind ActionKind (LAUNCH/TEARDOWN/RESET/BLOCK/NOOP) with NO transition whitelist, NO guarded rollback, NO script transitions, NO on_fail/advance auto-chaining, and NO per-column prompt/profile. The concurrency CAP slot model is the most insidious: reserve_slot/release_slot were faithfully ported into fs_store.py (and release IS called by teardown), but reserve_slot is NEVER called in the launch path — tick.py/LaunchAction never check the cap, so the daemon launches unbounded agents. The QUEUE (deferred launch on cap-full + reaper dequeue drain) is a documented stub (_drain_queue is empty; nothing ever writes a queue marker). interactive_only per agent column is unmodeled (replaced by a single global unattended_hours window). The per-launch append-only audit log (dispatch.jsonl) has no equivalent. _finalize_left_stage (✅) IS present and faithfully ported (8.1.e). The reaper's block-with-retry-relaunch (RETRY_LIMIT=1) is also gone. Net: faithful on the pivot and on sticky/teardown/reset/finalize, but the runner is a substantially simplified orchestrator — cap enforcement, queue, transition whitelist+rollback, script transitions, on_fail/advance, and the move-rate-limit park-in-Blocked are all missing or inert.

### STATE management — PoC kanbanmate/state.py (Store) + engine/cap.py vs NEW ports/store.py (StateStore Protocol) + adapters/store/fs_store.py (FsStateStore) + core/antiloop.py + app/tick.py (PersistedState)
The state slice was substantially re-shaped, not faithfully ported. NEW keeps the load-bearing, pivot-justified pieces: atomic state writes (temp+os.replace), corrupt-file-tolerant iteration (list_running), the slot cap (reserve_slot/release_slot under flock+O_EXCL, ported from engine/cap.py), the agent-advance breadcrumb (with a deliberate, documented re-key from content-node-id to issue-number), touch_heartbeat no-resurrection, and the kill-switch read. Several PoC persistence concerns are intentionally erased by the n8n→polling pivot and are correctly pivot_intended: the `processed/` delivery-idempotency cache (claim_delivery/already_processed/mark_processed/_delivery_key + path-traversal containment), the `botmoves/` on-disk TTL dedup ledger, the `inflight/` advisory lock (replaced by single-instance daemon flock), and the `prune()` TTL GC of those caches. HOWEVER several genuine feature_losses / behaviour_changes survive the pivot rationale: (1) the per-ticket move RATE LIMIT and the recent-move dedup are now PURE IN-MEMORY (core/antiloop.py, threaded via PersistedState) and NEVER persisted — on every daemon restart the rate-limit history, the recent-bot-move dedup, AND the diff baseline (columns_by_item) are all reset to empty, so a restart makes every card "first contact" and the runaway-loop rate backstop forgets its history; the PoC persisted moves/ and columns/ on disk. DESIGN §5/§6 sanctions the restart-rebuild posture for the diff baseline, but the rate-limit ledger losing durability is an undocumented weakening of the §6 runaway backstop. (2) The default rate limit silently changed from the PoC's transition default to a hard-coded 10/hour in core. (3) The `bookkeeping` flag on bot moves (the explicit anti-double-session-revert / anti-incoherent-rollback tag, persisted for audit) is gone entirely. (5) purge_ticket's exhaustive multi-dir teardown collapses into release_slot, which only removes state/, slots/, advances/ — the PoC additionally purged columns/, queue/, moves/ (issue- and item-keyed), botmoves/, retries/; some of those dirs no longer exist in NEW, but the queue marker and the move-history purge have no analogue. (6) No queue PERSISTENCE: queue_dir() is gone and _drain_queue is an explicit Phase-1 stub. (7) end_session's idle-transition semantics moved into the bin shim, not the store. None of these (except the diff-baseline durability) is justified by the n8n removal.

### engine adapters (worktree / tmux / perms / reaper / cap / locks / scripts)
Mixed fidelity. The low-level mechanical seams port faithfully: worktree create/remove(--force)/discover-branch (PoC engine/worktree.py → adapters/workspace/worktree.py), tmux launch/is_alive/kill (engine/tmux.py → sessions.py), the universal deny-list + defaultMode pinning + PostToolUse heartbeat hook (engine/perms.py → adapters/perms.py), and the flock locks + atomic cap reservation (engine/locks.py + cap.py → fs_store.py) are all present and largely verbatim — plus NEW adds a non-root guard and interval.py (next_sleep) is correctly wired into the daemon loop. BUT four genuine feature_losses unrelated to the n8n→polling pivot are present: (1) the concurrency CAP is reserve_slot dead code — fs_store.reserve_slot exists but is NEVER called by LaunchAction/tick/wiring, and the queue/drain is an explicit Phase-1 stub, so unlimited agents can launch concurrently (PoC enforced it on every launch at runner.py:706 and queued when full); (2) ensure_clone is entirely gone — no git init / remote add / fetch / credential-helper token isolation (a PoC SECURITY behaviour, tested), so the clone is assumed to pre-exist; (3) the run_transition_script / script-only mechanical transition path (engine/scripts.py + check-pr-ready.sh / check-merge-ready.sh) has no counterpart — there is no RUN_SCRIPT ActionKind and the two check scripts are not ported; (4) the reaper lost both the tmux-session-gone block-trigger (NEW only checks heartbeat TTL) and the RETRY_LIMIT relaunch-once-before-block behaviour that DESIGN §8.3 line 307 explicitly promises.
