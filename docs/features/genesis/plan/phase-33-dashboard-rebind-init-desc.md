# Phase 33 — Dashboard rebind + init short-desc

Two operator-observed fixes after a fresh board re-seed. (1) The rolling status-update state under `~/.kanban/status/` (`update_id` + `body_hash`) is board-wide and carried no project binding, so after the registry was re-pointed at a NEW project the stale `body_hash` suppressed the first post and the stale `update_id` pointed at the OLD board — no dashboard appeared on the new board until a manual purge. The fix persists the bound `project_id` alongside the markers (a new `status/project_id` file in `StatusUpdateStateMixin`, exposed via `get_status_project_id`/`set_status_project_id` on the `StateStore` port) and, in `report_status`, compares the stored binding to the live `deps.project_id`: on a mismatch (or never-bound marker) it drops the stale id+hash, re-binds to the current project, and falls through to a fresh create — while a matching binding preserves the on-change discipline (update-by-id, hash-suppressed no-op). (2) `kanban init` now sets a default project short description ("Kanban orchestrated by KanbanMate — autonomous Claude agents launched per transition (<repo>)") via a new `update_project_description(project_id, short_description)` seam on the `Seeder` port + `GithubClient` (a `node{shortDescription}` idempotency read + an `updateProjectV2` mutation, both behind the timed `_graphql` seam); the write is skipped when the project already has a non-empty description so an operator-authored one is never clobbered and a re-run of init is a no-op. Tests: store project_id round-trip/clear; reporter rebind forces a fresh create vs same-project keeps on-change; init wiring asserts the default description is set; client builder/mutation set-when-empty and skip-when-set.
