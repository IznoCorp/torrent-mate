bind every future wave (S2ŌĆōS7). This is the authoritative reference for the

## Overview

- **TorrentMate** is the application name. `personalscraper` is the code name
  (the Python engine/package). The frontend is referenced as **TorrentMateUI**.
- **S1 (tm-shell)** delivers the shell + auth + WebSocket foundation ŌĆö a
  headless FastAPI server serving the React SPA behind Caddy, with a
  Redis-bridged real-time event stream and an installable PWA.
- **S2ŌĆōS7** add pipeline control, maintenance dashboards, config editing,
  scraping, registry panels, and acquisition pages on these rails ŌĆö no
  per-wave re-architecture.

**Key design decisions:**

- The engine is **fully synchronous**, single-process-per-command. There is no
  HTTP server outside the `personalscraper web` daemon process.
- **Only the WS relay is async** ŌĆö `ws/relay.py` uses `redis.asyncio` for the
  stream-read loop and fan-out. All REST handlers are plain `def`; FastAPI
  runs them in its threadpool, calling the sync domain directly.
- **AppContext boundary rule**: `_build_app_context(config, settings)` in
  `personalscraper/commands/web.py` is the shared composition root. The
  `create_app(config, settings)` factory receives the already-built context and
  stores it on `app.state` for route and dependency access.
- The sync engine is never imported into async code paths beyond the uvicorn
  boundary.

## Auth

**Credentials** ŌĆö single-user. The username lives in `config/web.json5`
(`web.username`). The password is stored as a scrypt hash in the
`WEB_PASSWORD_HASH` environment variable, generated by `personalscraper web
set-password`. Format: `scrypt$N$r$p$salt$hash`.

**Session token** ŌĆö JWT HS256, signed with `WEB_JWT_SECRET` (generate with
`python -c "import secrets; print(secrets.token_urlsafe(32))"`). The cookie:

| Attribute | Value                                                      |
| --------- | ---------------------------------------------------------- |
| Name      | `tm_session`                                               |
| HttpOnly  | `True` (inaccessible to JS)                                |
| SameSite  | `Strict`                                                   |
| Secure    | Controlled by `web.cookie_secure` ŌĆö `True` in prod (HTTPS) |
| Path      | `/`                                                        |
| Max-Age   | `web.session_ttl_hours * 3600` (default 720 h = 30 days)   |

**Endpoints:**

| Method | Path               | Auth    | Response                                                                   |
| ------ | ------------------ | ------- | -------------------------------------------------------------------------- |
| POST   | `/api/auth/login`  | Public  | 204 + `Set-Cookie` on success; 401 on bad creds; 429 after 5 failures/60 s |
| POST   | `/api/auth/logout` | Guarded | 204 + cookie-clearing `Set-Cookie`                                         |
| GET    | `/api/auth/me`     | Guarded | `{"username": "izno"}`                                                     |

**Login rate limiting** ŌĆö per-client sliding window: 5 failed attempts within
60 seconds triggers a `429 Too Many Requests` response (non-blocking ŌĆö no
`time.sleep`). Success resets the window. The client key is derived from the
real IP via `X-Forwarded-For` when behind the Caddy reverse proxy.

**Guard perimeter** ŌĆö `require_session` is a FastAPI dependency that reads the
`tm_session` cookie, decodes the JWT, and verifies the `sub` claim matches
`config.web.username`. It is applied to **all `/api/*` routes except:**

- `GET /api/health` ŌĆö the liveness probe (DESIGN ┬¦4.4: health stays public)
- `POST /api/auth/login` ŌĆö the login endpoint itself

`/api/version` is **inside** the guard perimeter ŌĆö unauthenticated requests
receive `401`.

**Fail-closed secrets** ŌĆö an empty `WEB_JWT_SECRET` or missing
`WEB_PASSWORD_HASH` causes login to return `401` (with a one-time warning log),
never a `500`.

**CSRF posture** ŌĆö same-origin SPA + `SameSite=Strict` + JSON-only bodies.
S2+ mutating routes will additionally require `X-Requested-With: TorrentMate`.

## WebSocket Protocol

**Endpoint**: `GET /ws/events` (authenticated via the `tm_session` cookie).

### Handshake

The server **accepts the socket first**, then reads the cookie and validates
the JWT. On failure it closes with custom code `4401`. This accept-then-close
sequence is load-bearing: closing _before_ accept would produce an opaque `1006`
in the browser, making the client's terminal-close logic dead in production.

### Message shapes

After successful auth, the server sends exactly one hello:

```json
{ "type": "ws.hello", "data": { "build_commit": "<sha>" } }
```

Then, for each event that arrives via Redis:

```json
{"id": "<stream-id>", "type": "<EventClassName>", "data": {ŌĆ”}}
```

The event type is resolved from `envelope["_type"]` (the single source of truth
written by `event_to_envelope`). The redundant `type` field in the stream
entry is a fallback only.

### Keep-alive

Every 30 seconds of client silence the server sends:

```json
{ "type": "ws.ping" }
```

The client replies with any text frame (treated as a pong). Missed pings ŌåÆ
disconnect ŌåÆ client reconnects with backoff.

### Replay on reconnect

The client passes `?last_id=<stream-id>` on connect. The server uses
`XRANGE (last_id +` (exclusive lower bound) to replay all missed entries in
order before entering live fan-out.

A `_ReplayGuard` forwarder is registered **before** replay runs so events
emitted during the replay window are buffered (not lost), then flushed and
deduped against the replay high-water id once replay completes.

## Redis Event Relay

### Producer side (`personalscraper/subscribers/redis_stream.py`)

`RedisEventPublisher` subscribes to the base `Event` class on the in-process
EventBus. The callback enqueues to a bounded `queue.Queue(maxsize=1000)` and
returns immediately ŌĆö a daemon worker thread drains the queue and performs the
blocking `XADD` calls.

**Wiring sites**: `build_redis_publisher(event_bus, web_config)` is called from
`_build_app_context` (the shared composition root). It is gated on
`web.enabled` ŌĆö when `False`, `None` is returned and no publisher is wired.
Callers: `commands/pipeline.py` (the `run` command), `commands/watch.py` (the
Watcher daemon), and acquisition commands.

**Fail-soft contract** (same pattern as the Telegram notifier):

- Redis down ŌåÆ warn once (`redis_publish_failed`), drop events, never raise,
  never block the pipeline.
- Queue full ŌåÆ warn once (`redis_publish_queue_full`), drop the event.

**Stream key**: `web.stream_key` (default `"personalscraper:events"`).
**MAXLEN**: `web.stream_maxlen` (default `10000`), approximate trimming.

### Consumer side (`personalscraper/web/ws/relay.py`)

One `redis.asyncio` reader task per web process: `XREAD BLOCK 5000` from `$`,
fanning out every new entry to all connected WebSockets via
`ConnectionRegistry.broadcast()`.

Each `send_json` is bounded by `BROADCAST_SEND_TIMEOUT` (1 second) ŌĆö a stalled
client is dropped without blocking fan-out for everyone else.

Poison entries (bad JSON, missing `data` key) are logged once and skipped;
`last_id` always advances so the loop can never wedge on a malformed entry.

## PWA

TorrentMate is an **installable Progressive Web App** targeting Android, iOS,
and desktop.

### Install prompting

- **Android / desktop**: captures the `beforeinstallprompt` event and surfaces
  an in-app ┬½ Installer TorrentMate ┬╗ button. Dismissals are persisted to
  `localStorage` and remembered across sessions.
- **iOS / iPadOS**: detects Safari outside standalone mode via UA sniffing
  (`iPhone` / `iPad` in the UA, or `MacIntel` platform + `maxTouchPoints > 0`
  for iPadOS 13+ disguising as desktop Safari). Shows a native instruction
  sheet: Partager ŌåÆ ┬½ Sur l'├®cran d'accueil ┬╗.

### Service Worker (`vite-plugin-pwa` / Workbox)

- **`registerType: 'prompt'`** ŌĆö the `needRefresh` signal from
  `useRegisterSW` drives the update toast. (An earlier design specced
  `autoUpdate` ŌĆö under that mode `needRefresh` never fires and the update
  toast is dead code.)
- **Precache**: app shell only (`index.html`, JS/CSS bundles, icons, fonts).
- **Runtime**: `/api/*` and `/ws` are `NetworkOnly` in every mode ŌĆö never
  cached. Navigation requests fall back to the precached `index.html`.

### Update discipline (all installs converge)

1. SW update check on load, on `visibilitychange`, and every 15 minutes.
2. `/api/version` is polled; if the served `build_commit` differs from the
   baked `__BUILD_COMMIT__`, `registration.update()` is forced.
3. New SW detected ŌåÆ `needRefresh` ŌåÆ toast ┬½ Nouvelle version install├®e ŌĆö
   rechargementŌĆ” ┬╗ ŌåÆ `updateServiceWorker(true)` ŌåÆ `skipWaiting()` +
   `clients.claim()` + old caches deleted ŌåÆ single auto-reload.
4. Every installed client converges to the new build; no stale clients.

### Deploy scripts

**`scripts/deploy.sh`** (prod only ŌĆö refuses anything but a clean `main` synced
with `origin/main`):

1. Guard: branch == `main`, tree clean, `HEAD == origin/main`, prod venv exists.
2. `cd frontend && npm ci && TM_BUILD_COMMIT=<sha> npm run build`.
3. `rsync -a --delete frontend/dist/ ŌåÆ personalscraper/web/static/`.
4. Stamp `BUILD_COMMIT` with the exact served SHA.
5. `pip install -e .` into the prod venv.
6. `pm2 restart torrentmate-web`.
7. Post-check: `curl http://127.0.0.1:8710/api/health` ŌåÆ 200 (retries up to
   15 ├Ś 2 s = 30 s).

**`scripts/deploy-staging.sh`** (serves the current branch of the staging
clone, refuses dirty trees):

1. Guard: tree clean, staging venv exists.
2. Bake `"branch @ sha"` as `TM_BUILD_COMMIT` for the SPA and `BUILD_COMMIT`
   (byte-for-byte identical ŌĆö prevents perpetual phantom updates from the PWA
   comparing "sha" against "branch @ sha").
3. Same build/install/restart flow as prod, targeting port 8711.
4. Post-check against `127.0.0.1:8711`.

### Autodeploy poller

**`scripts/autodeploy-poll.sh`** ŌĆö PM2 app `torrentmate-autodeploy`, 60-second

- Prod (`~/deploy/torrentmate`): `git fetch origin main` ŌåÆ if `main` advanced

## REST Contract Conventions (binding for S2ŌĆōS7)

These conventions are recorded now and are **binding on every future wave**:

1. **Typed contract pipeline**: Pydantic request/response models ŌåÆ OpenAPI
   schema (`make openapi` ŌåÆ `frontend/openapi.json`, committed) ŌåÆ
   `openapi-typescript` ŌåÆ `frontend/src/api/schema.d.ts` (generated, committed).
   CI fails on drift (`git diff --exit-code`).

2. **Reads = `GET`** ŌĆö plain `def` handlers, no special guard beyond auth.

3. **Writes (S2+) = `POST`** ŌĆö must acquire the **same `pipeline.lock`** as the
   Watcher (single trigger authority, DESIGN ┬¦4.6). The EventBus stays
   observe-only; write commands flow through the same lock that serialises
   `watch`-driven pipeline runs.

4. **Errors**: RFC-ish `{"detail": "..."}` with proper HTTP status codes.
   `401` everywhere unauthenticated. `429` on login rate-limit.

5. **CSRF for mutating routes**: all S2+ mutating routes must require
   `X-Requested-With: TorrentMate` in addition to the session cookie.

6. **Guard mount point**: in `create_app`, a `guarded_api` router carries
   `Depends(require_session)`. All future `/api/*` routers mount inside it ŌĆö
   the auth perimeter is a single dependency, not scattered per-route.

7. **Frontend data layer**: TanStack Query client with typed `fetcher<Path>`
   built on the generated OpenAPI types, `credentials: 'include'` for cookie
   auth, global `401` handler ŌåÆ redirect `/login`.

## Pipeline control (S2 ŌĆö `pipe-control`)

S2 turns the `/pipeline` screen into the operator control deck. All mutating
routes require the session cookie **and** `X-Requested-With: TorrentMate`; every
write goes through the same `pipeline.lock` as the Watcher (single trigger
authority ŌĆö no parallel writer; the EventBus stays observe-only).

### REST endpoints (`/api/pipeline`)

| Method | Path                 | Guard         | Semantics                                                                                                                                                                                                                                                 |
| ------ | -------------------- | ------------- | --------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------- |
| POST   | `/run`               | session + XRW | Spawn `personalscraper run --no-console --trigger-reason=web [--dry-run]` (detached). `409` if the lock is held. Returns `202 {run_uid}`. The `run_uid` is injected to the subprocess via `PERSONALSCRAPER_RUN_UID`, so it matches the run's history row. |
| POST   | `/pause`             | session + XRW | Create the `pipeline.pause` sentinel. The engine pauses at the next step boundary.                                                                                                                                                                        |
| POST   | `/resume`            | session + XRW | Remove the `pipeline.pause` sentinel.                                                                                                                                                                                                                     |
| POST   | `/kill`              | session + XRW | SIGTERM the run pid (read from the lock file); clear the pause sentinel; the run finalizes its history row as `killed`.                                                                                                                                   |
| POST   | `/watcher`           | session + XRW | `{enabled}` ŌĆö `false` writes the `watcher.paused` sentinel (the watch loop no-ops); `true` removes it. Distinct from pausing a running pipeline.                                                                                                          |
| GET    | `/status`            | session       | `{state, run_uid?, step?, paused, watcher_enabled, pid?}`. | GET    | `/history`           | session       | `?limit&offset&sort` (`started_at`, `-started_at`, `duration`, `-duration`) ŌåÆ `{runs: RunSummary[], total}`.                                                                                                                                              |

### Pause mechanics ŌĆö two distinct levers

- **Pause pipeline**: the engine calls a cooperative checkpoint **between the 9
  steps**; if the `pipeline.pause` sentinel (in `paths.data_dir`) exists it polls
  until the sentinel clears or SIGTERM arrives. Granularity is the step boundary ŌĆö
  an in-flight step is never interrupted by pause (only Kill stops mid-step).
  Emits `PipelinePaused` / `PipelineResumed` (relayed to the WS like all events).
- **Pause watcher**: the `watcher.paused` sentinel stops the daemon from
  _auto-starting_ new runs. It does not touch a run already in progress.

### Run-history store ŌĆö `pipeline_run` table

The indexer DB gains a `pipeline_run` table (migration `011`). Each run writes its
own record (fail-soft ŌĆö a history-write error never aborts the run):

| Column       | Type         | Notes                                                     |
| ------------ | ------------ | --------------------------------------------------------- |

with environment variables `PERSONALSCRAPER_RUN_UID`,

### Validate-before-write pipeline

Both `POST /validate` (dry-run) and `PUT /files/{name}` (pre-write gate) call
`validate_candidate()` in `personalscraper/conf/loader.py`. The function:

1. Reads the on-disk overlay files from the config directory.
2. Substitutes the candidate dict for the targeted file **in memory** (no
   filesystem mutation during validation).
3. Runs the full merge pipeline (`merge_overlays()`) followed by Pydantic
   `Config.model_validate()`.
4. Returns `(validated_config, warnings)` on success; raises
   `ConfigValidationError` (with the underlying `pydantic.ValidationError` as
   `__cause__`) on failure.

The route handler extracts Pydantic `loc`/`msg`/`type` entries from the
`ValidationError` and returns them as a `422` JSON array ŌĆö the frontend maps
them to form fields via `flattenLocToPath()` (dot-joined path convention).

### Atomic write + backups

`PUT /files/{name}` serialises writes through a module-level `threading.Lock`
(routes are sync handlers in the FastAPI threadpool, so `threading.Lock` is the
correct primitive). The entire write sequence runs inside the lock ŌĆö the SHA-256
precondition check and `validate_candidate()` call are also inside the critical
section, so the compare-and-write is atomic (TOCTOU-safe):

1. **Precondition**: the request `base_sha256` is compared against the current
   file digest. A mismatch returns `412` before any mutation.
2. **Validate**: `validate_candidate()` runs against the candidate values in
   memory (no filesystem mutation). On `ConfigConflictError` ŌåÆ `422`; on
   `ConfigLoadError` for another declared overlay ŌåÆ `409`.
3. **Backup**: if the file exists, copy it to
   `config/.backups/{name}.{utc_microsecond}.json5` (microsecond timestamps
   prevent collisions when saves land within the same second).
4. **Prune**: keep the 10 most recent backups per file name; older ones are
   deleted.
5. **Atomic write**: write the new content to a `tempfile.mkstemp` in the config
   directory, `flush()` + `fsync()`, then `os.replace()` onto the target path
   (atomic on the same filesystem). On any failure the temp file is unlinked.
6. **Header comment**: every rewritten file gets a generated first line:
   `// Written by TorrentMate config editor <ISO-8601-utc> ŌĆö hand-written comments are not preserved.`

### Restart-impact classification

`RESTART_IMPACT` (module-level dict in `personalscraper/web/routes/config.py`)
maps each top-level config key to whether changing it requires a web process
restart:

| Impact  | Keys                                                                                                                                                                                                                                                                                                                                  |
| ------- | ------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------- |
| `True`  | `web`, `paths`, `indexer`                                                                                                                                                                                                                                                                                                             |
| `False` | `acquire`, `anime_rule`, `categories`, `category_rules`, `config_version`, `cross_seed`, `custom_categories`, `disks`, `fuzzy_match`, `genre_mapping`, `ingest`, `library`, `metadata`, `notify`, `process_clean`, `providers`, `ranking`, `scraper`, `sort`, `staging_dirs`, `thresholds`, `torrent`, `tracker`, `trailers`, `watch` |

Unknown keys default to `True` (fail-safe: if a key is not in the map, assume a
restart is needed). An architecture test (phase-gate 4.5) asserts that every
`Config.model_fields` key has an entry in `RESTART_IMPACT`, so the dict can
never silently miss a newly-added config key.

### Role gating

The `PERSONALSCRAPER_WEB_ROLE` environment variable gates all write endpoints:

- **Unset or `"prod"`**: writes are allowed (normal operation).
- **`"staging"`**: `PUT /files/{name}`, `PUT /secrets`, and
  `POST /restart-web` all return `403 {"detail": "read-only"}`. Read endpoints
  (`GET /schema`, `GET /files`, `GET /files/{name}`, `GET /status`,
  `GET /secrets`) remain available ŌĆö the editor is fully explorable but
  non-mutating.

The `PERSONALSCRAPER_PM2_NAME` environment variable enables the restart
endpoint:

- **Set** (e.g. `"torrentmate-web"`): `POST /restart-web` spawns a detached
  subprocess that sleeps 0.5 s (so the `202` response flushes first), then runs
  `pm2 restart <name>`. The subprocess runs in a new session (`start_new_session=True`)
  with stdout/stderr redirected to `<tmpdir>/torrentmate-restart-web.log`
  (truncated per spawn, opened with `O_NOFOLLOW`) so a failed pm2 invocation
  leaves a trace. If the log file cannot be opened, the error is logged and the
  subprocess falls back to `DEVNULL` ŌĆö the restart still proceeds.
- **Unset**: `POST /restart-web` returns `404 {"detail": "restart not configured"}`.

Both variables are set per-PM2-app in `ecosystem.config.js` ŌĆö not in `.env`.

**Async-restart semantics (known limitation).** The endpoint answers `202`
_before_ the detached restart runs ŌĆö it confirms the restart was _scheduled_,
not that it _succeeded_. A restart that then fails (pm2 not on the subprocess
`PATH`, wrong app name, pm2 daemon down) is **not** surfaced to the caller: the
web connection never drops and the UI shows "restart scheduled". The only trace
is the `torrentmate-restart-web.log` file plus the `config_restart_spawned`
warning log line. A future enhancement could have the frontend poll `/status`
(the boot-hash snapshot resets after a real restart) and warn if the process is
still serving the old config after a timeout; this is intentionally not built
in the S4 write-only + restart-badge model.

### Secrets editor

The secrets subsystem (`personalscraper/conf/envfile.py`) provides two shared
utilities consumed by the API routes:

- **`read_env_catalog(env_example_path)`** ŌĆö parses `.env.example` into a
  `{KEY: description}` catalog. Keys match `^[A-Z][A-Z0-9_]*=`. Descriptions
  are the contiguous `#` comment lines above each key (section-rule lines
  starting with `# ŌöĆŌöĆ` **reset the comment accumulator** ŌĆö they describe the
  following key group but are never attached to a specific key; a blank line
  also resets the accumulator).
- **`write_env_keys(keys, env_path)`** ŌĆö atomically upserts `{KEY: value}`
  pairs into `.env`. Existing lines with matching keys are replaced in place;
  all other lines (comments, blanks, unrelated keys) are preserved. New keys
  are appended. Atomic via same-directory temp file + `os.replace`.

The API enforces:

- **Write-only**: `GET /secrets` returns `is_set` flags only ŌĆö values are never
  read from `.env` or included in the response.
- **Catalog guard**: `PUT /secrets` rejects keys not declared in `.env.example`
  with `422 {"unknown_keys": [...]}`. Values are **never** echoed in the error
  detail. The route logs `config_secrets_write` with sorted key names only (no
  values).
- **`.env.example` catalog** (as of 0.43.0): `QBIT_USERNAME`, `QBIT_PASSWORD`,
  `TMDB_API_KEY`, `TVDB_API_KEY`, `TRAKT_CLIENT_ID`, `TELEGRAM_BOT_TOKEN`,
  `TELEGRAM_CHAT_ID`, `HEALTHCHECK_URL`, `YOUTUBE_API_KEY`,
  `YOUTUBE_COOKIES_FILE`, `YOUTUBE_COOKIES_FROM_BROWSER`, `OMDB_API_KEY`,
  `WEB_PASSWORD_HASH`, `WEB_JWT_SECRET`. (`PERSONALSCRAPER_WEB_ROLE` and
  `PERSONALSCRAPER_PM2_NAME` are PM2-only env vars ŌĆö they are documented in
  `.env.example` as comments but not in the catalog because they are set per
  PM2 app, not in `.env`.)

### Known limits

- **JSON5 comments are not preserved** on rewritten files. The web editor writes
  standard JSON5 (via the `json5` library) with a generated header comment. Hand-written
  inline comments in config files are lost on the first web edit. This matches the
  `init-config` precedent (template files carry comments; runtime config files
  are machine-maintained). Operators who rely on inline comments should keep
  notes in `config.example/` (the tracked template) or version-control their
  `config/` directory.
- **Git-tracked config files get dirty on web edits**. If `config/` is under
  version control, every web edit produces a modified working tree. This is
  accepted as an audit trail (D4 decision in DESIGN.md ┬¦Appendix) ŌĆö the
  `config/.backups/` directory provides an additional recovery layer independent
  of git.
