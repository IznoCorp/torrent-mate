# TorrentMate Web UI — Reference

Architecture, auth, WebSocket protocol, Redis event relay, PWA mechanics, local
development setup, deploy runbook, and the typed REST contract conventions that
bind every future wave (S2–S7). This is the authoritative reference for the
shipped S1 foundation (`feat/tm-shell`, ticket #158).

## Overview

- **TorrentMate** is the application name. `personalscraper` is the code name
  (the Python engine/package). The frontend is referenced as **TorrentMateUI**.
- **S1 (tm-shell)** delivers the shell + auth + WebSocket foundation — a
  headless FastAPI server serving the React SPA behind Caddy, with a
  Redis-bridged real-time event stream and an installable PWA.
- **S2–S7** add pipeline control, maintenance dashboards, config editing,
  scraping, registry panels, and acquisition pages on these rails — no
  per-wave re-architecture.

## Architecture

```
┌─────────────── producer processes (unchanged, sync) ───────────────┐
│ personalscraper run / watch / grab / …                             │
│   EventBus ──> RedisEventPublisher (new subscriber, fail-soft)     │
│                   └── XADD envelope → Redis Stream                 │
└────────────────────────────────────────────────────────────────────┘
                                   │ Redis Stream `personalscraper:events`
                                   ▼ (MAXLEN ~10k, ids = replay cursor)
┌─────────────── web process — `personalscraper web` ────────────────┐
│ uvicorn (async)                                                    │
│  FastAPI app (personalscraper/web/)                                │
│   ├─ /api/auth/*  /api/health  /api/version   (REST, Pydantic)     │
│   ├─ /ws/events   (auth-guarded; tail stream → fan-out + replay)   │
│   └─ static: TorrentMateUI build (SPA fallback) + BUILD_COMMIT     │
└────────────────────────────────────────────────────────────────────┘
                                   ▲ Caddy: tm.iznogoudatall.xyz (prod :8710)
                                   ▲        tm-staging.iznogoudatall.xyz (:8711)
Clients: browser / installed PWA (Android, iOS, desktop)
```

**Key design decisions:**

- The engine is **fully synchronous**, single-process-per-command. There is no
  HTTP server outside the `personalscraper web` daemon process.
- **Only the WS relay is async** — `ws/relay.py` uses `redis.asyncio` for the
  stream-read loop and fan-out. All REST handlers are plain `def`; FastAPI
  runs them in its threadpool, calling the sync domain directly.
- **AppContext boundary rule**: `_build_app_context(config, settings)` in
  `personalscraper/commands/web.py` is the shared composition root. The
  `create_app(config, settings)` factory receives the already-built context and
  stores it on `app.state` for route and dependency access.
- The sync engine is never imported into async code paths beyond the uvicorn
  boundary.


## Auth

**Credentials** — single-user. The username lives in `config/web.json5`
(`web.username`). The password is stored as a scrypt hash in the
`WEB_PASSWORD_HASH` environment variable, generated by `personalscraper web
set-password`. Format: `scrypt$N$r$p$salt$hash`.

**Session token** — JWT HS256, signed with `WEB_JWT_SECRET` (generate with
`python -c "import secrets; print(secrets.token_urlsafe(32))"`). The cookie:

| Attribute | Value                                                      |
| --------- | ---------------------------------------------------------- |
| Name      | `tm_session`                                               |
| HttpOnly  | `True` (inaccessible to JS)                                |
| SameSite  | `Strict`                                                   |
| Secure    | Controlled by `web.cookie_secure` — `True` in prod (HTTPS) |
| Path      | `/`                                                        |
| Max-Age   | `web.session_ttl_hours * 3600` (default 720 h = 30 days)   |

**Endpoints:**

| Method | Path               | Auth    | Response                                                                   |
| ------ | ------------------ | ------- | -------------------------------------------------------------------------- |
| POST   | `/api/auth/login`  | Public  | 204 + `Set-Cookie` on success; 401 on bad creds; 429 after 5 failures/60 s |
| POST   | `/api/auth/logout` | Guarded | 204 + cookie-clearing `Set-Cookie`                                         |
| GET    | `/api/auth/me`     | Guarded | `{"username": "izno"}`                                                     |

**Login rate limiting** — per-client sliding window: 5 failed attempts within
60 seconds triggers a `429 Too Many Requests` response (non-blocking — no
`time.sleep`). Success resets the window. The client key is derived from the
real IP via `X-Forwarded-For` when behind the Caddy reverse proxy.

**Guard perimeter** — `require_session` is a FastAPI dependency that reads the
`tm_session` cookie, decodes the JWT, and verifies the `sub` claim matches
`config.web.username`. It is applied to **all `/api/*` routes except:**

- `GET /api/health` — the liveness probe (DESIGN §4.4: health stays public)
- `POST /api/auth/login` — the login endpoint itself

`/api/version` is **inside** the guard perimeter — unauthenticated requests
receive `401`.

**Fail-closed secrets** — an empty `WEB_JWT_SECRET` or missing
`WEB_PASSWORD_HASH` causes login to return `401` (with a one-time warning log),
never a `500`.

**CSRF posture** — same-origin SPA + `SameSite=Strict` + JSON-only bodies.
S2+ mutating routes will additionally require `X-Requested-With: TorrentMate`.

## WebSocket Protocol

**Endpoint**: `GET /ws/events` (authenticated via the `tm_session` cookie).

### Handshake

The server **accepts the socket first**, then reads the cookie and validates
the JWT. On failure it closes with custom code `4401`. This accept-then-close
sequence is load-bearing: closing _before_ accept would produce an opaque `1006`
in the browser, making the client's terminal-close logic dead in production.

### Message shapes

After successful auth, the server sends exactly one hello:

```json
{ "type": "ws.hello", "data": { "build_commit": "<sha>" } }
```

Then, for each event that arrives via Redis:

```json
{"id": "<stream-id>", "type": "<EventClassName>", "data": {…}}
```

The event type is resolved from `envelope["_type"]` (the single source of truth
written by `event_to_envelope`). The redundant `type` field in the stream
entry is a fallback only.

### Keep-alive

Every 30 seconds of client silence the server sends:

```json
{ "type": "ws.ping" }
```

The client replies with any text frame (treated as a pong). Missed pings →
disconnect → client reconnects with backoff.

### Replay on reconnect

The client passes `?last_id=<stream-id>` on connect. The server uses
`XRANGE (last_id +` (exclusive lower bound) to replay all missed entries in
order before entering live fan-out.

A `_ReplayGuard` forwarder is registered **before** replay runs so events
emitted during the replay window are buffered (not lost), then flushed and
deduped against the replay high-water id once replay completes.

## Redis Event Relay

### Producer side (`personalscraper/subscribers/redis_stream.py`)

`RedisEventPublisher` subscribes to the base `Event` class on the in-process
EventBus. The callback enqueues to a bounded `queue.Queue(maxsize=1000)` and
returns immediately — a daemon worker thread drains the queue and performs the
blocking `XADD` calls.

**Wiring sites**: `build_redis_publisher(event_bus, web_config)` is called from
`_build_app_context` (the shared composition root). It is gated on
`web.enabled` — when `False`, `None` is returned and no publisher is wired.
Callers: `commands/pipeline.py` (the `run` command), `commands/watch.py` (the
Watcher daemon), and acquisition commands.

**Fail-soft contract** (same pattern as the Telegram notifier):

- Redis down → warn once (`redis_publish_failed`), drop events, never raise,
  never block the pipeline.
- Queue full → warn once (`redis_publish_queue_full`), drop the event.

**Stream key**: `web.stream_key` (default `"personalscraper:events"`).
**MAXLEN**: `web.stream_maxlen` (default `10000`), approximate trimming.

### Consumer side (`personalscraper/web/ws/relay.py`)

One `redis.asyncio` reader task per web process: `XREAD BLOCK 5000` from `$`,
fanning out every new entry to all connected WebSockets via
`ConnectionRegistry.broadcast()`.

Each `send_json` is bounded by `BROADCAST_SEND_TIMEOUT` (1 second) — a stalled
client is dropped without blocking fan-out for everyone else.

Poison entries (bad JSON, missing `data` key) are logged once and skipped;
`last_id` always advances so the loop can never wedge on a malformed entry.

## PWA

TorrentMate is an **installable Progressive Web App** targeting Android, iOS,
and desktop.

### Install prompting

- **Android / desktop**: captures the `beforeinstallprompt` event and surfaces
  an in-app « Installer TorrentMate » button. Dismissals are persisted to
  `localStorage` and remembered across sessions.
- **iOS / iPadOS**: detects Safari outside standalone mode via UA sniffing
  (`iPhone` / `iPad` in the UA, or `MacIntel` platform + `maxTouchPoints > 0`
  for iPadOS 13+ disguising as desktop Safari). Shows a native instruction
  sheet: Partager → « Sur l'écran d'accueil ».

### Service Worker (`vite-plugin-pwa` / Workbox)

- **`registerType: 'prompt'`** — the `needRefresh` signal from
  `useRegisterSW` drives the update toast. (An earlier design specced
  `autoUpdate` — under that mode `needRefresh` never fires and the update
  toast is dead code.)
- **Precache**: app shell only (`index.html`, JS/CSS bundles, icons, fonts).
- **Runtime**: `/api/*` and `/ws` are `NetworkOnly` in every mode — never
  cached. Navigation requests fall back to the precached `index.html`.

### Update discipline (all installs converge)

1. SW update check on load, on `visibilitychange`, and every 15 minutes.
2. `/api/version` is polled; if the served `build_commit` differs from the
   baked `__BUILD_COMMIT__`, `registration.update()` is forced.
3. New SW detected → `needRefresh` → toast « Nouvelle version installée —
   rechargement… » → `updateServiceWorker(true)` → `skipWaiting()` +
   `clients.claim()` + old caches deleted → single auto-reload.
4. Every installed client converges to the new build; no stale clients.

### Static assets served

| Path                    | Content-Type                | Notes                        |
| ----------------------- | --------------------------- | ---------------------------- |
| `/manifest.webmanifest` | `application/manifest+json` | TorrentMate PWA manifest     |
| `/sw.js`                | `text/javascript`           | Workbox service worker       |
| `/assets/*`             | hashed static files         | Vite build output            |
| `/*` (fallback)         | `text/html`                 | `index.html` for SPA routing |

## Local Development Setup

### Backend

```bash
# Set the password hash and JWT secret (one-time):
personalscraper web set-password

# Or set manually in .env:
#   WEB_PASSWORD_HASH=<scrypt hash from the command above>
#   WEB_JWT_SECRET=<output of python -c "import secrets; print(secrets.token_urlsafe(32))">

# Run the web daemon (foreground):
personalscraper web

# For local dev without HTTPS, the config must have cookie_secure: false.
# This is the default in config.example/web.json5 on the dev checkout.
```

The server binds `127.0.0.1:8710` by default (configurable via `--host` and
`--port` CLI flags, or in `config/web.json5`).

### Frontend

```bash
cd frontend
npm ci
npm run dev        # Vite dev server with HMR — proxies /api + /ws to :8710
```

For a production-like build:

```bash
cd frontend
npm run build      # writes to ../personalscraper/web/static/
```

`personalscraper web` refuses to start if `static/index.html` is missing and
`web.dev_mode` is `false` — this prevents serving a half-deployed app.

### Dependencies

  `PyJWT`.
- **Dev**: `httpx` (TestClient), `fakeredis` (stream-capable, for relay tests).

## Deploy Runbook

Two deploy clones mirror KanbanMate's model — per-clone venvs for isolation
from the dev editable install.

### Clones

| Role    | Path                    | Tracks    | PM2 app                   | Port |
| ------- | ----------------------- | --------- | ------------------------- | ---- |
| Prod    | `~/deploy/torrentmate`  | `main`    | `torrentmate-web`         | 8710 |
| Staging | `~/staging/torrentmate` | `staging` | `torrentmate-web-staging` | 8711 |

Both clones share the **real config directory** via
`PERSONALSCRAPER_CONFIG=/Users/izno/dev/PersonalScraper/config` in the PM2
environment. S1 is read-only, so staging against real data is safe (KanbanMate
"no test board" rule).

Each clone has its own venv at `~/deploy/torrentmate-venv` /
`~/staging/torrentmate-venv` (overridable via `TM_VENV` / `TM_STAGING_VENV`).

### Deploy scripts

**`scripts/deploy.sh`** (prod only — refuses anything but a clean `main` synced
with `origin/main`):

1. Guard: branch == `main`, tree clean, `HEAD == origin/main`, prod venv exists.
2. `cd frontend && npm ci && TM_BUILD_COMMIT=<sha> npm run build`.
3. `rsync -a --delete frontend/dist/ → personalscraper/web/static/`.
4. Stamp `BUILD_COMMIT` with the exact served SHA.
5. `pip install -e .` into the prod venv.
6. `pm2 restart torrentmate-web`.
7. Post-check: `curl http://127.0.0.1:8710/api/health` → 200 (retries up to
   15 × 2 s = 30 s).

**`scripts/deploy-staging.sh`** (serves the current branch of the staging
clone, refuses dirty trees):

1. Guard: tree clean, staging venv exists.
2. Bake `"branch @ sha"` as `TM_BUILD_COMMIT` for the SPA and `BUILD_COMMIT`
   (byte-for-byte identical — prevents perpetual phantom updates from the PWA
   comparing "sha" against "branch @ sha").
3. Same build/install/restart flow as prod, targeting port 8711.
4. Post-check against `127.0.0.1:8711`.

### Autodeploy poller

**`scripts/autodeploy-poll.sh`** — PM2 app `torrentmate-autodeploy`, 60-second
loop:

- Prod (`~/deploy/torrentmate`): `git fetch origin main` → if `main` advanced
  → `git pull --ff-only` → `deploy.sh`.
- Staging (`~/staging/torrentmate`): `git fetch origin staging` → if `staging`
  advanced → `git reset --hard origin/staging` (staging may be rebased) →
  `deploy-staging.sh`.
- `--once` flag for testing; `AUTODEPLOY_INTERVAL` env overrides the loop
  interval.
- Fail-soft per cycle — one failed pass never kills the loop.

### Caddy reverse proxy

Blocks at `/opt/homebrew/etc/Caddyfile` (operator-applied):

```caddy
https://tm.iznogoudatall.xyz {
    import tls_config
    reverse_proxy localhost:8710
}
https://tm-staging.iznogoudatall.xyz {
    import tls_config
    reverse_proxy localhost:8711
}
```

WebSocket proxying is native to Caddy `reverse_proxy` — no extra directive
needed. Caddy-level basicauth is NOT required (the app enforces its own
JWT-cookie auth on `/api/*` and the WS handshake).

In `ecosystem.config.js`:

| App                       | Script / interpreter  | Notes                                      |
| ------------------------- | --------------------- | ------------------------------------------ |
| `torrentmate-web`         | `personalscraper web` | cwd: `~/deploy/torrentmate`, prod venv     |

### BUILD_COMMIT stamping

- **Prod**: `BUILD_COMMIT` contains the full SHA of the deployed commit.
- **Staging**: `BUILD_COMMIT` contains `"<branch> @ <sha>"`.
- Both match the `TM_BUILD_COMMIT` baked into the SPA at build time
  (`vite.config.ts` → `define __BUILD_COMMIT__`), so the PWA's version poll
  compares byte-for-byte.
- `GET /api/version` returns `{"version": "<package>", "build_commit": "<stamp>"}`.

## REST Contract Conventions (binding for S2–S7)

These conventions are recorded now and are **binding on every future wave**:

1. **Typed contract pipeline**: Pydantic request/response models → OpenAPI
   schema (`make openapi` → `frontend/openapi.json`, committed) →
   `openapi-typescript` → `frontend/src/api/schema.d.ts` (generated, committed).
   CI fails on drift (`git diff --exit-code`).

2. **Reads = `GET`** — plain `def` handlers, no special guard beyond auth.

3. **Writes (S2+) = `POST`** — must acquire the **same `pipeline.lock`** as the
   Watcher (single trigger authority, DESIGN §4.6). The EventBus stays
   observe-only; write commands flow through the same lock that serialises
   `watch`-driven pipeline runs.

4. **Errors**: RFC-ish `{"detail": "..."}` with proper HTTP status codes.
   `401` everywhere unauthenticated. `429` on login rate-limit.

5. **CSRF for mutating routes**: all S2+ mutating routes must require
   `X-Requested-With: TorrentMate` in addition to the session cookie.

6. **Guard mount point**: in `create_app`, a `guarded_api` router carries
   `Depends(require_session)`. All future `/api/*` routers mount inside it —
   the auth perimeter is a single dependency, not scattered per-route.

7. **Frontend data layer**: TanStack Query client with typed `fetcher<Path>`
   built on the generated OpenAPI types, `credentials: 'include'` for cookie
   auth, global `401` handler → redirect `/login`.

## Pipeline control (S2 — `pipe-control`)

S2 turns the `/pipeline` screen into the operator control deck. All mutating
routes require the session cookie **and** `X-Requested-With: TorrentMate`; every
write goes through the same `pipeline.lock` as the Watcher (single trigger
authority — no parallel writer; the EventBus stays observe-only).

### REST endpoints (`/api/pipeline`)

| Method | Path                 | Guard         | Semantics                                                                                                                                                                                                                                                 |
| ------ | -------------------- | ------------- | --------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------- |
| POST   | `/run`               | session + XRW | Spawn `personalscraper run --no-console --trigger-reason=web [--dry-run]` (detached). `409` if the lock is held. Returns `202 {run_uid}`. The `run_uid` is injected to the subprocess via `PERSONALSCRAPER_RUN_UID`, so it matches the run's history row. |
| POST   | `/pause`             | session + XRW | Create the `pipeline.pause` sentinel. The engine pauses at the next step boundary.                                                                                                                                                                        |
| POST   | `/resume`            | session + XRW | Remove the `pipeline.pause` sentinel.                                                                                                                                                                                                                     |
| POST   | `/kill`              | session + XRW | SIGTERM the run pid (read from the lock file); clear the pause sentinel; the run finalizes its history row as `killed`.                                                                                                                                   |
| POST   | `/watcher`           | session + XRW | `{enabled}` — `false` writes the `watcher.paused` sentinel (the watch loop no-ops); `true` removes it. Distinct from pausing a running pipeline.                                                                                                          |
| GET    | `/status`            | session       | `{state, run_uid?, step?, paused, watcher_enabled, pid?}`. State: `idle`, `running`, or `paused` — derived from the lock file, sentinel files, and the latest `pipeline_run` row.                                                                         |
| GET    | `/history`           | session       | `?limit&offset&sort` (`started_at`, `-started_at`, `duration`, `-duration`) → `{runs: RunSummary[], total}`.                                                                                                                                              |
| GET    | `/history/{run_uid}` | session       | One run with per-step timings → `RunDetail`; `404` if unknown.                                                                                                                                                                                            |

### Pause mechanics — two distinct levers

- **Pause pipeline**: the engine calls a cooperative checkpoint **between the 9
  steps**; if the `pipeline.pause` sentinel (in `paths.data_dir`) exists it polls
  until the sentinel clears or SIGTERM arrives. Granularity is the step boundary —
  an in-flight step is never interrupted by pause (only Kill stops mid-step).
  Emits `PipelinePaused` / `PipelineResumed` (relayed to the WS like all events).
- **Pause watcher**: the `watcher.paused` sentinel stops the daemon from
  _auto-starting_ new runs. It does not touch a run already in progress.

The indexer DB gains a `pipeline_run` table (migration `011`). Each run writes its
own record (fail-soft — a history-write error never aborts the run):

## Maintenance dashboard (S3 — `maint-dash`)

S3 exposes the ~20 `library-*` Typer CLI commands as a catalog of web-UI
maintenance actions, backed by a registry-driven form generator and a detached
subprocess runner. All read endpoints are `GET` with the standard session guard;
the single write endpoint (`POST …/run`) additionally requires
`X-Requested-With: TorrentMate`. Route contract: `docs/features/maint-dash/DESIGN.md` §4.

| ------ | ------------------------------------------ | ------------- | ------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------ |

| GET    | `/api/maintenance/locks`                   | session       | `LocksResponse` — pipeline lock status, in-flight maintenance run info.                                                                                                                                                                                                                    |

### Extended run-history — `?kind=` filter

S2's `/api/pipeline/history` gains a `?kind=` query parameter:

| Value           | Behavior                                              |
| --------------- | ----------------------------------------------------- |
| `pipeline`      | Returns only pipeline runs (`kind='pipeline'`).       |
| `maintenance`   | Returns only maintenance runs (`kind='maintenance'`). |
| `all` (default) | Returns both. Invalid values → `400`.                 |

`GET /api/pipeline/history/{run_uid}` now also returns `kind`, `command`,
`options_json`, and `output_tail`. The `RunSummary` and `RunDetail` Pydantic
models gained `kind` + `command` (RunDetail also `options_json` + `output_tail`).

**Unified store.** Migration `012` adds `kind` (`'pipeline' | 'maintenance'`,
default `'pipeline'`), `command`, `options_json`, and `output_tail` columns to
the `pipeline_run` table. Pipeline and maintenance runs share one table — the
frontend `RunHistoryTable` on `/maintenance` filters via `?kind=maintenance`.

### Action registry & runner

**Registry** (`personalscraper/web/maintenance/registry.py`). `REGISTRY` holds 25
`MaintenanceAction` entries — one per `library-*` Typer command. Each entry:

| Field          | Type                                       | Notes                                                 |
| -------------- | ------------------------------------------ | ----------------------------------------------------- |
| `id`           | `str`                                      | Matches the CLI command name.                         |
| `title`        | `str` (FR)                                 | Human-readable label.                                 |
| `description`  | `str` (FR)                                 | One-line summary of what the action does.             |
| `category`     | `query\|scan\|repair\|clean\|analyze\|fix` | Drives the grouped catalog UI.                        |
| `risk`         | `ro\|write\|destructive`                   | Drives the risk badge and dry-run gating.             |
| `long_running` | `bool`                                     | Shown in the UI before launch.                        |
| `dry_run`      | `supported\|unsupported`                   | Whether the CLI accepts `--dry-run`.                  |
| `options`      | `list[ActionOption]`                       | Curated, typed form fields generated by the frontend. |

A test asserts `set(REGISTRY.keys()) == set(registered library-* commands)`, so
the registry can never drift from the CLI surface.

**Runner** (`personalscraper/web/maintenance/runner.py`). The POST handler spawns
a detached subprocess:

```
python -m personalscraper.web.maintenance.runner
```

with environment variables `PERSONALSCRAPER_RUN_UID`,
`PERSONALSCRAPER_MAINT_COMMAND`, `PERSONALSCRAPER_MAINT_OPTIONS_JSON` (canonical
form), and `PERSONALSCRAPER_MAINT_DRY_RUN`. The runner:

1. Inserts a `pipeline_run` row with `kind='maintenance'`.
2. Runs the real `library-*` CLI as a child process.
3. Streams each output line to the Redis stream as a `maintenance.run_log`
   envelope (keyed by `run_uid` — relayed to the WebSocket with no protocol
   change).
4. Keeps a 64 KiB `output_tail` ring buffer.
5. Finalizes the row (`success` / `error`) with `output_tail` persisted.

**Dry-run-first UX.** Destructive/write actions require a successful dry-run before
the Apply button unlocks. The backend enforces this with a `428` when the last
matching dry-run is older than 30 minutes or absent. Editing any option
re-locks Apply (the previous dry-run is invalidated). The runner has a
per-command `_DRY_RUN_STYLE` table handling the two CLI conventions (`--dry-run`
vs `--apply` absence = dry-run).

### Frontend

`/maintenance` (replaces the S1 `ComingSoon` stub) renders four panels:

- **DisksPanel** — per-disk space gauges, mount state, category breakdown.
- **LocksPanel** — pipeline lock status + in-flight maintenance run PID/uptime.
- **IndexHealthPanel** — DB table sizes, repair-queue depth, last scan age.
- **RunHistoryTable** — reused from S2, filtered to `?kind=maintenance` with
  kind chips.

Below the panels, the **ActionCatalog** groups actions by category with risk
badges (`ro` / `write` / `destructive`). Selecting an action opens the
**ActionForm** — fields are generated from the action's `options` list. The flow
enforces dry-run-first: after a successful dry-run the Apply button unlocks;
editing any option or receiving a backend `428` re-locks it. Once launched, the
**RunOutput** panel shows the live log via the WS-fed `RunLogFeed` (reused from
S2) with `output_tail` as the durable fallback from the history detail endpoint.
